# CodexGuideTalk 特效设计规格

## 目标

为 `episodes/CodexGuideTalk/` 中约 3 分 19 秒的口播视频制作一套 Remotion 特效。整体编排以该期 `script.md` 的组件建议为准，并以 `talk.srt` 的实际语义时间点重新对齐。

本期不使用外部截图或录屏。原脚本中依赖素材的画面统一替换为抽象解释型视觉，不模拟真实产品界面，不把个人经验、产品行为或未经核实的数据伪装成事实证据。

## 已确认的制作边界

- 使用现有 C13、C16、C21、C23、C24、C26、C27，不新增公共组件。
- 不修改公共组件的表现、样式、动效、布局、行为或 props。
- 不使用渐变。
- 一段口播只承担一个主要视觉任务。
- 每段最多一个主组件和两个辅助标注。
- 人物居中时，普通组件不得进入人物安全区。
- 正文不得进入字幕安全区。
- 本阶段只做到关键帧 QC 和 Remotion Studio 网页预览，不生成视频文件。
- 只有用户在网页预览后明确要求，才执行低清预览或高清渲染。

## 视觉方向

沿用项目现有的高密度科技面板语言：

- 黑白信息层级配酸绿、青色、黄色或橙色单色强调。
- 全屏流程段允许较高信息密度；人物居中段只在左右边缘放置摘要。
- C24 负责章节标题，C23 负责数字与冲击结论，C21 负责步骤和对比，C26 负责右侧辅助标注。
- C13 负责测试队列、规则执行、HTML 交接和任务拆分等抽象流程。
- C16 只用于结尾的简短总结。
- C27 负责 `talk`、`push-in`、`speaker-left`、`speaker-right`、`pip-right` 与 `content-full` 等镜头切换。

## 分镜与组件设计

具体起止点允许在配置时按字幕边界微调，不能跨越语义段落。

### 1. 开场：使用痛点与收益

时间：0:00–0:13

- 主任务：建立“痛点、效率、Token、其他 Agent 也适用”的主题。
- 组件：C24 左侧标题“Codex 使用痛点”，C26 右侧辅助“省 Token / 提效率”。
- 镜头：先 `talk`，关键词出现时轻微 `push-in`。
- 限制：不提前展示四个技巧的细节。

### 2. 痛点一：过度测试

时间：0:13–0:35

- 0:13–0:24：C23 左侧先显示“干活 3 分钟”，再翻转为“测试 10 分钟”；C26 右侧显示“时间被吃掉”。
- 0:24–0:35：C13 全屏抽象执行面板，任务气泡依次出现“跑测试 / 截图检测 / 继续检查”，表现测试链路扩张。
- 镜头：人物段保持 `talk`，进入 C13 时切 `content-full` 或 `pip-right`。
- 证据边界：“3 分钟 / 10 分钟”作为口播观点的视觉转述，不附来源卡，不呈现为统计结论。

### 3. 方法一：限制深度测试

时间：0:35–0:47

- C24 左侧显示“写进自定义指令”。
- C13 抽象规则执行面板显示“读取规则 → 必要检查 → 结束”，并让多余测试节点退出。
- C26 右侧收束“够用就停”。
- 不伪造 `AGENTS.md` 截图或产品界面。

### 4. 痛点二：额度与追求目标

时间：0:47–1:29

- 0:47–0:58：C24 左侧章节标题“第二个痛点：额度”，C26 右侧显示“集中火力”。
- 0:58–1:04：C23 突出“1%”，作为个人经验叙述的关键数字。
- 1:04–1:18：C21 `steps` 依次强调“开追求目标 / 开完全访问 / 安排大任务”。
- 1:18–1:29：C13 抽象任务队列表现持续推进，C26 显示“集中用完”。
- 镜头：章节段 `talk`，步骤段 `speaker-right`，任务推进段 `pip-right`。
- 证据边界：不得画成官方承诺；画面中明确使用“个人用法”或“个人经验”标签。

### 5. 痛点三：审美与反复调整

时间：1:29–1:46

- C24 左侧章节标题“第三个痛点：审美”。
- C21 `compare` 显示“解决问题：强 / 审美：一般”。
- C13 全屏参数面板用“字体 / 间距 / 配色”三组反复变化的任务气泡表现返工。
- C26 右侧辅助“Token 继续消耗”。
- 不生成虚假的真实项目截图。

### 6. 方法三：网页端先定 HTML

时间：1:46–2:09

- C13 全屏抽象交接流程表现“网页 GPT → 调整样式 → HTML → Codex 实现”。
- C21 `steps` 在人物镜头中补充“先调样式 / 导出 HTML / 交给 Codex”。
- 镜头按 `speaker-right → pip-right → content-full → speaker-left` 推进，呼应原脚本的内容主次变化。
- HTML 文件使用抽象文件卡，不伪造真实网页端对话。

### 7. 两个好处

时间：2:09–2:24

- C24 左侧显示“两个好处”。
- C21 `compare` 显示“反复猜样式 / 完整 HTML”。
- C23 或 C26 分别收束“少占 Codex 额度”和“实现更稳”。
- 一次只高亮一个好处，避免同时抢读。

### 8. 技巧四：拆分大型任务

时间：2:24–3:00

- 2:24–2:34：C24 左侧章节标题“第四个技巧：拆任务”，C21 显示“网页端拆分 / 多会话执行”。
- 2:34–2:45：C21 `compare` 显示“一个长对话 / 多个短会话”。
- 2:45–3:00：C13 全屏抽象上下文面板。长上下文持续增长，压缩动作后部分细节节点熄灭，再切换为多个独立任务舱。
- C26 右侧先显示“消耗增长”，再显示“别漏细节”。
- 镜头以 `speaker-left → content-full → pip-right` 为主。
- 证据边界：额度增长、速度变慢和细节丢失均作为口播观点展示，不使用无来源图表或真实数据坐标。

### 9. 结尾

时间：3:00–3:18.8

- C24 左侧显示“不只适用于 Codex”。
- C26 右侧显示“Agent 都能用”。
- C16 依次打出“规则写清楚 / 任务安排明白 / 多干正事”。
- 镜头回到 `talk`，结尾轻微 `push-in`，不再重复前文面板。

## 数据与配置

需要在 `episodes/CodexGuideTalk/` 中补齐该项目既有 episode 结构：

- `episode.json`：视频、字幕、scene 和 shots 配置。
- `storyboard.md`：按实际 SRT 时间轴记录分镜。
- `asset-manifest.json`：只登记现有口播视频和字幕，不声明缺失的外部素材。
- `sources.json` 与 `sources.md`：保持空来源集合，并注明本期未使用证据型外部素材。
- `assets-needed.md`：记录用户已确认采用抽象视觉，不再等待外部素材。

所有 scene 都需包含稳定 ID、时间、track、kind、stageMode、slot、content、assetIds、sourceRefIds、status 和 notes。`shots` 使用帧编号并与 30 fps 对齐。

## QC 与验收

### 自动检查

- 运行 episode 配置校验。
- 运行 `npm run typecheck`。
- 运行 `npm run lint`。
- 运行相关 smoke test；不做深度测试。

### 关键帧 QC

至少检查：

- 开场左右摘要是否避开人物。
- C13 全屏段是否为抽象解释画面且不侵入字幕区。
- C21/C23/C24/C26 的正文底部是否高于字幕安全区。
- 人物左右让位时，组件和人物是否互不遮挡。
- 四个章节转换点是否与 SRT 语义边界一致。
- 结尾 C16 是否在结束前完整退出或稳定停留。

### 用户验收

自动检查和关键帧 QC 通过后启动 Remotion Studio，提供本地网页预览地址。用户确认前不执行 `episode:preview`、`episode:render`、`remotion render` 或任何其他视频导出命令。
